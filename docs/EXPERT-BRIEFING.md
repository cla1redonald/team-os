# Team OS & the shared context layer — master expert briefing
*Synthesised from the research briefs in `docs/research/`. Every external claim is sourced there; honesty flags carried through.*

## 1. The thesis (say this in the first 90 seconds)
Every team's AI is a **brilliant amnesiac**. It's fluent, but it doesn't know *your* metrics, *your* processes, *your* customers, *your* decisions. Engineers quietly solved this — they have git, `CLAUDE.md`/`AGENTS.md`, and now formats to hand agents curated context. **Everyone else didn't.** That's why AI adoption stalls outside engineering: not a model problem, a **shared-context problem**. A **Team OS** is the fix — the curated, agent-readable knowledge layer a team's AI runs on. Google just shipped a standard for it (OKF); the pattern is spreading function by function. I'll stand one up live.

## 2. The mental model — four layers (your whiteboard)
| Layer | What it is | Canonical tool/format | If missing |
|---|---|---|---|
| **Instructions** | how the agent should behave | AGENTS.md, CLAUDE.md, Cursor/Windsurf rules | agent ignores your conventions |
| **Knowledge** | curated facts about *your* domain | **OKF** (LLM-wiki), llms.txt | agent is generic / confidently wrong ← **the gap outside eng** |
| **Memory** | what happened before / evolving state | Mem0, Letta, Zep; **Beads** (task memory) | agent has "50 First Dates" amnesia |
| **Tools/Retrieval** | live capabilities & data | MCP; Agent Skills (progressive disclosure) | agent can't reach real systems |

One line: *Instructions = the staff handbook. Knowledge = the company wiki. Memory = their notebook. Tools = their access badge.* The instructions layer is near-commoditised in *authoring* (converging on AGENTS.md and SKILL.md) though team-wide *distribution* is an emerging market (cross-tool skill-sync platforms); tools has a winner (MCP); memory is a battleground; **the knowledge layer was the gap — that's what OKF and a Team OS fill.**

**The four layers are the anatomy; the boot sequence is the order of adoption.** Most companies' AI is still the DOS era — every person hand-loads the printer drivers into every conversation. The fix is the same move computing made forty years ago, in the same order: (0) bare metal → (1) **the kernel** boots — canonical concepts, definitions, ownership; small, authored, owned — (2) **system calls** — numbers resolve through governed definitions, agents never touch raw SQL — (3) **users and permissions** — scoped views over one kernel — (4) **the indexer** — retrieval engines like enterprise search find everything but own nothing — (5) **the journal** — who changed what definition, when — (6) **userland** — agents finally get to act, sandboxed, because everything below exists. Each stage makes the next one safe; pain reorders the middle, never the ends. *An agent without a kernel is a confident amnesiac with hands.*

## 3. OKF — the knowledge layer, standardised (3 weeks old)
Google's **Open Knowledge Format** (v0.1, 12 June 2026, `GoogleCloudPlatform/knowledge-catalog`, Apache-2.0): a directory of markdown files, **one "concept" per file**, only `type` required in frontmatter, concepts cross-linked into a **graph**; optional `index.md`/`log.md`. It's just markdown — forkable, git-hostable, **editable by a non-engineer**. It formalises the "LLM-wiki" pattern into something portable and vendor-neutral.
- **The sceptic's line to pre-empt:** *"a standard, or just a folder?"* Answer: technically a folder with rules — the value isn't expressiveness, it's **convergence/interop**. One month in, don't claim victory; claim it's the first serious attempt to standardise the knowledge layer.
- **Why not just RAG?** RAG retrieves from messy source docs; OKF is *curated* knowledge authored for agents — higher signal, version-controlled, human-auditable.

## 4. Beads — the memory layer, and the freshness flex
Steve Yegge's **Beads** (`bd`): git-native **external memory for coding agents** — a DAG of issues with dependencies, `bd ready` returns unblocked work in ~10ms. Solves the **"50 First Dates"** amnesia. 25k stars, "100% vibe coded," seven rearchitectures in three weeks.
- **The detail that proves you're current:** Beads **migrated from SQLite+JSONL to Dolt.** Every launch-era write-up describes the old stack — cite Dolt and you're clearly first-hand.
- **The honest altitude:** Beads is *task* memory, not a knowledge base ("Confluence vs Jira — Beads is Jira"). It's the memory layer; OKF is the knowledge layer. Don't conflate them — that's the most common architecture mistake.

## 5. The proof outside engineering (your thesis, evidenced)
- **Foundational:** Hannah Stulberg (**DoorDash**) — shared repo as team OS, *"no technical background required."* Teresa Torres & David Wascha — *"the markdown files ARE the system."*
- **Flagship:** **Pawel Huryn** (Product Compass, ~131k subs) — `pm-brain` ("Second Brain for PMs, made of markdown," no vector DB), `pm-skills`, `swarm-protocol` ("no Jira, just state sync"). A full three-layer Team OS for a non-eng function, open-source, at scale.
- **By function:** Marketing has named builders (Emily Kramer/MKT1, Elaine Zelby/Tofu's *"ICP, personas, messaging"* triplet, Animalz's style-guide-as-8-agents). Recruitment: Guillermo Flor's hiring OS where *onboarding comes from the same scorecard that hired the person.*
- **The killer finding for the room:** eng, **PM and marketing** have named individuals hand-rolling markdown Team OSes; **sales, HR and finance** have *vendors* selling "the context layer," not DIY practitioners. **The layer is furthest along exactly where people can hand-roll markdown, and stalls where governance and non-technical users dominate.** Your demo is the bridge — it hands the hand-rollable pattern to the stuck functions.

## 6. Build vs buy (the CTO debate you'll get)
A hand-curated OKF Team OS does by hand, for a small team, what **Glean / M365 Copilot / Atlassian Rovo / Notion / Dust / Guru** productise at scale.
- **DIY wins:** portability / zero lock-in (plain text in git — enterprise indexes are proprietary and often non-exportable; you *rent* retrieval), ~$0 cost, curation quality for <50-person teams, agent-neutral via MCP.
- **Enterprise wins:** connector breadth, **permission-aware retrieval at scale** (the #1 reason to buy — a DIY OKF has no per-user ACL), governance (SSO/SCIM, audit, residency).
- **The trigger to buy is capability, not cost:** buy when you need *enforced permissions* and *live-system reach*, which arrives before the dollars hurt.
- **Memory ≠ knowledge:** curated knowledge (OKF) is authored/governed; memory (Mem0/Letta/Zep/Beads) is derived/mutable. You need both.

**The verified 2026 packaging pattern (pricing pages, July 2026 — date everything, these change quarterly):** MCP/agent access is commoditising down-market (Augment bundles it in a $100/mo base plan; dbt hosts MCP on all plans), while the paywall is consistently **identity, compliance, and deployment control** — SSO/SCIM, CMEK, ISO 42001/27001, self-hosting are Enterprise-gated everywhere. And the new lock-in is **metering mechanics, not data formats**: dbt exhausting AI credits blocks *all* tools through its hosted MCP server; Augment's headline price includes only $100 of usage then bills at LLM list price +40%; Sourcegraph is enterprise-only from $16K with MCP gated behind it; Palantir's ontology export is negotiable only before signature. **Scrutinise the meter in the contract, not just the export button.**

**"Should we build our own MCP server?" — the trichotomy.** "MCP" conflates three things, and the build-vs-buy answer differs per thing:
1. **The protocol** — a standard. You adopt standards; nobody builds their own POSIX.
2. **The infrastructure** — gateways, registries, auth, audit. Undifferentiated plumbing in a fragmented ~30-vendor market with production-proven open source. Adopt or buy; never build.
3. **MCP servers over your own systems** — an MCP server is an *interface to something*. When the something is yours — your product, your data, your kernel — you build the interface, because nobody else can. In OS terms: the hardware vendor writes the driver for its own device; it just doesn't write its own USB stack. The proof is universal: dbt, Sourcegraph, Augment, Cube, and Glean all *built* MCP servers over their own products while telling customers to *adopt* the protocol.

Plus the refinement that elevates it: **tool design is curation.** One vendor benchmark found a ~100x token difference between two MCP toolkits for the *same* SaaS on identical queries — payload shaping, not connector count, is the differentiator. Which tools you expose and what shape their answers take encodes what questions your org asks — so your MCP tool surface is part of the *kernel*, not the wiring. The line: **"Build MCP servers over what you own. Never build the gateway. Treat tool definitions as curated context."**

## 9. Q&A defence — if someone asks…
- *"How is this different from CLAUDE.md / agents.md files in our repos and profiles?"* → It isn't different in kind — those files are the existence proof, and you can absolutely put truths (not just instructions) in one; non-engineers author them in user profiles every day. The difference is scope and governance. **Scope:** a CLAUDE.md is *personal and per-tool* — one person's note, loaded whole by one vendor's product, invisible to every other agent on the team. The kernel is *shared and addressable* — one register in team space, one concept per file, pointed at selectively by anyone's agent, any vendor. **Governance:** nothing keeps a CLAUDE.md true — no owner per fact, no journal, no supersession, no patrol; a meaning-rich CLAUDE.md is a kernel trapped in a private note. One-liner: "CLAUDE.md is your note to one tool; the kernel is the team's register for every tool — it's where your CLAUDE.md goes when it grows up."
- *"Isn't this just a wiki?"* → Yes, pointed at agents and standardised for interop. The novelty is convergence, not the format.
- *"How do you stop people changing the kernel without permission?"* → The register defines the requirements — every change carries an owner's signature, lands in the journal, and is tamper-evident (attributed, diffable, revertible). **Your stack supplies the enforcement** — the kernel is a *folder*, not a repo, and this is deliberate: business users don't use git, and requiring it is exactly how the knowledge layer stalled outside engineering. Three equal shapes: a Docs-native shop gets Doc ownership + Suggesting mode; a team already living in repos gets branch protection + CODEOWNERS generated from the ownership map (machine-enforced merge gates, if you want them); a small or solo estate can skip platforms entirely and let the *loop* be the gate — agents write proposals to an inbox, only owners write to the register, and the journal + patrol catch bypasses. In every shape, agents are propose-only by construction: an agent that can silently edit definitions can move your targets. No knowledge store is tamper-*proof* against a determined insider — the kernel is tamper-*evident*, which is the real upgrade over the wiki everyone edits and nobody audits. The follow-up ("but a shared folder means anyone *can* edit it") deserves its own concession: files don't defend themselves — which is why the shape is **read-everyone, write-owners** (shared folders have ACLs, managed by the IdP you already run; the board pack is "just files" too), proposals go to an open inbox rather than the register, version history attributes every change, and a register change with no journal entry is itself a patrol finding. The gate scales with stakes: for regulated definitions choose Suggesting-only or repo gates — because the stakes demand it, not the pattern. And the tripwire has two clocks: the *batch* version is the patrol's `unjournalled-change` check (a concept dated after the journal's newest entry bypassed the loop); the *event* version fires on the write itself — a push with no linked PR (one API call to check), or a Drive/SharePoint file-modified flow with no covering suggestion — and alerts the owner with the revert link in hand. Same comparison, run at write-time instead of patrol-time.
- *"What does the proposal inbox look like for a team — not a solo vault?"* → Every stack already has a native "awaiting someone's decision" queue; the inbox maps straight onto it. **Repo shape:** the inbox *is* the PR queue — proposal = PR, CODEOWNERS routes it to the owner, approve = the election, review comments = amendments, merge commit = the journal, patrol in CI on the same event. **Docs shape:** Suggesting mode is the inbox for small changes (Docs notifies the owner natively; Accept/Reject is the attributed signature); a `Proposals/` folder with @-mentions for bigger ones; the patrol digest is *delivered* per owner via Slack/Teams/email (kernel-keeper's notify recipes). **Chat-native:** the agent posts the before/after to a channel with an approve action; the owner taps; the agent applies the change and writes the journal line. Honesty about who has this today: **engineers already run a queue (PRs); most business teams do not** — approvals live in meetings and email threads. So for them the queue arrives *with* the kernel, and it must be radically light or adoption dies (the git lesson again). Cadence matters as much as weight — **the loop needs a trigger AND a schedule**: events notify immediately (a proposal lands or the register changes → the owner hears the same hour, not Friday — a proposal that waits a week teaches proposers to stop proposing); the *schedule* exists only for what time alone reveals (staleness sweeps, silent journals); and **pending elections re-surface until answered** — an unanswered proposal is itself a finding. The owner's experience stays light: a same-day nudge per proposal, one periodic digest, reply-to-elect. The invariant everywhere: a queue, a notification that finds the owner, a signature, a journal line — build the lightest version the owner will actually answer.
- *"Didn't the base model already know most of that?"* → The domain, yes — that's why the ungrounded answer sounds competent. But it's a generic framework that ends by asking *you* for data; grounded, it's a specific diagnosis in your language ending in your runbook's action, and everyone gets the *same* answer. Then run the canary (the escalation codename) — no base model can produce that.
- *"Why not RAG over everything?"* → Everything-buckets return confident answers from stale/contradictory sources. Curated + verified beats retrieved-from-mess. You need both.
- *"Doesn't OKF lock me into Google?"* → Opposite — Apache-2.0, vendor-neutral, plain markdown; serve it over MCP to Claude/Gemini/ChatGPT alike.
- *"Adoption?"* → One month old. Don't over-index on stars. It's the first serious knowledge-layer standard.
- *"Permissions?"* → The honest DIY gap — no per-user ACL. Above ~100 seats or regulated data, that's your trigger to buy.
- *"Beads?"* → Task memory, git-native, now Dolt-backed; adopt the *pattern*, pin the version; unproven outside eng.
- *"We built our own MCP server — are you saying we shouldn't have?"* → Opposite. You built the driver for your own device — that's the pattern working; every vendor does exactly the same over their own product. What you shouldn't build is the *gateway* (fragmented market, production-proven OSS) or a second-hand server to someone else's SaaS. See §6's trichotomy.
- *"Your metric file is just prose and a CSV."* → Correct — which is why it defers to a governed definition (`definitions/*.yml`); in production that resolves through the semantic layer over MCP. The number is compiled, not retrieved. That's Beat 3 of the demo.
- *"Why isn't our data platform plus connectors enough?"* → Connectors give agents **access, not judgment**. The warehouse knows *what happened* — it has the `connection_attempts` table; it does not know what CSR means, who owns it, or that a single-bank drop means "them, not us." And the tribal rules — the escalation codename, "don't mass-retry," "that bank's Sunday dip is planned maintenance" — exist in **no system a connector can reach**; they were in someone's head until authored. Worse: every connector added amplifies the ten-answers problem — the agent can now read all ten conflicting decks instantly. Access multiplies context; only an owned, authored layer arbitrates it. Concession: connectors are stage 4 (buy them), and for numbers the data platform growing its own meaning layer *is* the semantic layer (stage 2) — the kernel is the same move for everything that isn't a number. *The platform knows what happened. The kernel knows what you mean.*
- *"But it already exists somewhere a connector can reach — why recreate it as a file?"* → **The kernel isn't a copy of your content; it's the register of which content is canonical.** Usually "it exists" means six versions exist in six places — and the fact that would resolve them (*which one counts*) exists nowhere; the kernel file is that election, not a recreation. Where something genuinely exists once and well, **point, don't copy**: four lines — canonical source, owner, last affirmed, anything-disagreeing-is-stale (see `processes/enterprise-client-onboarding.md` for the pattern). Think **DNS for meaning**: DNS doesn't duplicate the website, it authoritatively resolves the name. And drift reverses: without the register the sources drift from each other with no arbiter; with it there's one small, owned, journaled drift surface. Don't over-extend the DNS analogy: it covers the *resolution* half (authoritatively name what's canonical); the TTL equivalent is the affirm-or-amend cadence, and the patrol checks pointer *integrity* — whether the pointed-at source has *changed meaning* underneath a valid pointer is an unsolved check (semantic drift — the next deterministic check on the patrol's roadmap), and saying so beats stretching the analogy. Kernel files are thin by design: elections, pointers, and tacit rules — not content.
- *"Who actually does this work — isn't this a librarian?"* → Yes, and we've run this experiment before: the 1990s KM wave hired corporate librarians, and it died of **three** things — no reader, high capture cost, and no incentive to contribute. The agent-reader fixes the first and *changes* the other two: capture cost drops because the kernel is 25 files, not 25,000; and the incentive flips because the SME's own agent gives worse answers until they fix the entry — **rot becomes self-punishing**. The agent is the tireless reader that consults the register on every task — the librarian is back, and this time she has a customer. Shape matters: **librarian = registrar, not oracle**. Domain owners own the elections (finance owns what churn means); the librarian owns the register's health — structure, cross-links, staleness sweeps, chasing owners to affirm-or-amend. And most of that patrol is agent-automatable: human librarian for judgment, agent librarian for hygiene.
- *"We already share skills across the team — isn't that this?"* → A skill tells agents **how**; the kernel tells them **what's true**. Procedure register vs truth register — same substrate (portable markdown), different asset. Skill *distribution* is solved and buyable (cross-tool sync platforms — a driver, buy it), and **marketplace trust is arriving fast**: NVIDIA's Verified Agent Skills (May 2026) signs, scans and reviews third-party skills, after research found ~80% of one public registry's skills mismatched declaration vs behaviour. What that leaves open is the **org-internal register**: who owns *your* home-grown skills, elections between conflicting internal ones, supersession, staleness patrol. The notary exists for the marketplace; for your own library, most orgs already run the registrar *ad hoc* — a skills repo with review — without naming it. That's the point: like the kernel, internal skill governance is a **discipline, not a product** — own, elect, supersede, patrol. If you already do it in git, you've proved the pattern. *You need both registers, and both need owners.* And there's a third authored asset: **the roster** — agent definitions, each a *composition* of the other two (a scoped view of the kernel + skills + tool grants + a model). The scoped-views demo is a proto-roster: "the exec agent" and "the support agent" are roster entries. Sharing agents without a kernel is sharing confident amnesiacs — context boots first, but the estate is three registers. The razor in its cleanest form: **own everything authored — truth, procedure, roster; rent everything infrastructural — engines, gateways, runtimes.** (Agent *runtimes* are rented; agent *definitions* are owned.)
- *"Can't we just automate the librarian entirely?"* → About 80%, today — link-checking, staleness sweeps, pointer-vs-source drift, gap-mining for repeated unanswered questions, drafting proposed concepts, affirm-or-amend nudges. And that 80% is exactly the toil that killed KM 1.0. The 20% that must stay human is the **elections**: which definition counts is an authority problem, not a knowledge problem. Agents propose, never commit — the register's entire value is that a human with authority signed it; automate the signature and you've destroyed the asset (and built the poisoning vector). **You can automate the clerk, not the judge — the judgment being human is what the register is for.** And this isn't aspiration — the keeper's current version ships the drafting clerk: **the Scribe** turns patrol findings into drafted affirm-or-amend proposals (the owner's decision collapses to a yes/no), and **the Gap-Miner** sweeps a folder of working notes into new-concept proposals for truths no register concept covers. Propose-only is enforced, not asked: proposals are written to an inbox *outside* the register through a single write path with no route into it (an out-dir inside the bundle is a hard error), every output carries provenance (proposer, model, a prompt-identity hash, the evidence files), and proposal ids are deterministic so re-runs skip instead of spamming the inbox. Receipts — not a study, one register, my own — from the weekend before this talk: the miner surfaced 14 proposals; I elected 13 and **amended 10 of them before signing**. Pre-empt the flip ("77% needed rewriting — what did it save you?"): amend ≠ reject — the skeleton, evidence and journal line stood; I changed wording and thresholds, not whether the truth was real. The draft did the fetching; I did the deciding. What's deliberately not built yet is the third leg — the deterministic clerk that *applies* what the judge signs (plus drift-watching and answer-auditing); today the judge applies what they sign, and saying so beats claiming the loop closed.
- *"We're a Google/M365 shop — nobody writes local markdown or does pull requests."* → Markdown is the **wire format, not the user interface**, and git is one implementation of the journal, not the kernel's definition. Map the requirements onto your stack: a shared Drive/SharePoint folder = the canonical place; Doc owner + folder ACLs = the ownership map; **version history = the journal, automatically**; **Suggesting mode / approval flows = the review step non-engineers already use every day**; Drive/SharePoint permissions are already IdP-scoped; agents read it through connectors today; and Docs exports markdown natively — so the content stays owned and portable, which is what "own the kernel" means (own ≠ self-host). Three shapes, in adoption order: Docs-native → Docs-as-authoring with a sync-to-markdown system of record → the agent as the editor, where the SME says "update the definition," the owner approves in chat, and nobody asks where the files live — the same way nobody asks where their calendar's .ics files live. **Be precise about the trade:** Docs-native is the authoring *on-ramp*, not the governed end-state — Suggesting mode has no CODEOWNERS and version history is per-file; sync-to-markdown is the *option* for teams that want machine-enforced per-concept gates and clean diffs — a choice, not a maturity ladder, and the patrol never forces it: it walks any folder, git or not, including a synced Drive. The demo uses local files for the same reason the gateway is two folders: zero infrastructure to show the pattern.
